# Change History

This log tracks major decisions and milestones as the scaffold application evolves.

| Date | Change | Notes |

| 2026-06-08 | Guided PASTA risk analysis and publication workflow (004-guided-pasta-risk) | Extended the PASTA threat modeling workflow with structured stage-seven risk scoring and risk-workspace publication. New model: `PastaRiskConclusion` (`pasta_risk_conclusions` table) — one-to-one with `PastaFinding` (type `risk_conclusion`), storing `likelihood_score`, `impact_score`, `overall_score`, `treatment`, `publication_state` (`not_published` / `published` / `needs_revalidation`), optional `published_risk_id` FK into the risk workspace, `last_published_at`, `last_published_by_id`, and `publication_notes`. New `PastaPublicationState` enum and `PASTA_STAGE_GUIDANCE` constant (per-stage i18n keys for purpose / expected inputs / downstream outputs). New services: `compute_pasta_overall_score`, `apply_pasta_conclusion_scores`, `publish_pasta_conclusion_to_risk` (idempotent — republish refreshes the linked Risk row rather than creating a duplicate), and `export_pasta_findings_csv_with_scores` (extends the base CSV with score and publication columns). New routes: `pasta_risk_conclusion_edit` (GET/POST scoring form at `/<model_id>/pasta/findings/<finding_id>/risk`) and `pasta_publish_risk` (POST at `/<model_id>/pasta/findings/<finding_id>/publish-risk`). Stage guidance panels (purpose, expected inputs, downstream outputs) shown in `pasta_stage_form.html` for all seven stages. Risk dashboard (`risk/dashboard.html`) now surfaces a **PASTA source** badge with a link back to the originating conclusion for any risk that was published from a PASTA model. Publication state badges and Add/Edit Scores links surface in both `model_detail.html` and `pasta_stage_form.html` for stage-seven `risk_conclusion` findings. HTML/PDF export updated to show scored conclusions in stage-seven finding blocks. CSV export extended with `likelihood_score`, `impact_score`, `overall_score`, `risk_priority`, and `publication_state` columns. Migration `20260608_0002_pasta_risk_conclusions` creates the new table and backfills existing stage-seven `risk_conclusion` findings. Tests added across `test_threat_pasta_models.py`, `test_threat_pasta_routes.py`, `test_threat_pasta_workflow.py`, and `test_threat_pasta_exports.py`. Translations updated in both `en.json` and `nl.json`. |
| 2026-06-08 | PASTA threat modeling workflow (003-pasta-threat-modeling) | Added model-level PASTA threat modeling as a first-class workflow alongside STRIDE-LM. `ThreatModel` gains a `methodology` field and an optional `bootstrap_source_model_id` for one-way traceability. New models: `PastaStageRecord` (seven ordered stages per PASTA model), `PastaFinding` (per-stage findings with optional STRIDE-LM category annotations and asset links), `PastaFindingAssetLink`, `PastaFindingStrideCategoryLink`, `PastaFindingThreatScenarioLink`. Stage progression uses light gating (each stage unlocks when the previous stage has a summary). Editing an earlier stage triggers `needs_revalidation` on later completed stages. Threat-oriented findings (`THREAT`, `VULNERABILITY`, `ATTACK_PATH`) can generate or link standard `ThreatScenario` records for downstream traceability. A `bootstrap_pasta_from_stride` service creates a new PASTA model from an existing STRIDE-LM model (one-way, non-destructive). Exports branch by methodology: PASTA models produce a PASTA-specific HTML report and a findings-based CSV. All new functionality is covered by dedicated test modules: `test_threat_pasta_models.py`, `test_threat_pasta_routes.py`, `test_threat_pasta_workflow.py`, `test_threat_pasta_bootstrap.py`, `test_threat_pasta_mappings.py`, `test_threat_pasta_exports.py`, `test_threat_pasta_scenarios.py`. Migration `20260608_0001_pasta_threat_modeling` adds all new tables and columns. Existing STRIDE-LM models and exports are unaffected. |
| 2026-05-27 | Partial backup restore added | Administrators can now restore selected tables from `.db.gz`, `.db.gz.enc`, `.sql.gz`, and `.sql.gz.enc` backups without replacing the entire database. Includes inspection, conflict preview, skip-existing execution, per-table results, and audit logging. |
| 2026-04-10 | SSP boundary UX, controls picker overhaul, and risk controls sync (2026.4.6) | Three focused improvements: **(1) SSP authorization boundary UX** — The SSP edit page now surfaces a collapsible BIA-sourced boundary panel that lists the linked context's component names and interconnection system names as chip badges (blue for components, green for interconnections). A "Fill from BIA" button copies these items into the `authorization_boundary` textarea; a manual add-item input lets editors append individual entries without touching the textarea directly. When `authorization_boundary` is empty on a GET request the field is auto-populated from the BIA items. A `render_field_with_help` macro is now used throughout the edit form, and `description` (i18n) help text was added to all SSP, interconnection, control-entry, and POA&M form fields. ~39 new translation keys in `en.json` and `nl.json`. **(2) Controls picker redesign** — The flat `<select multiple>` for CSA controls on the threat scenario form and the risk edit form has been replaced with a searchable scrollable checkbox panel. A live filter input narrows items by label text; a count badge shows how many controls are selected; CSS `:has(input:checked)` highlights selected rows; unselecting all via the treatment toggle clears the checkboxes and resets the filter. **(3) Risk controls auto-sync** — The risk `edit` route now re-syncs the linked risk's controls list from the threat scenario on every load (matching the existing components re-sync introduced in 2026.4.5), keeping controls in lock-step with the threat model without manual intervention. Minor Docker restore Dockerfile fix included. |
| 2026-04-09 | Field-level encryption, SSP/POA&M deep integration, and risk dashboard refactor (2026.4.5) | Three main areas of work: **(1) Field-level encryption** — New `scaffold/core/encryption.py` introduces `EncryptedString`, `EncryptedJSON`, and `EncryptedBinary` SQLAlchemy `TypeDecorator` types backed by `cryptography.Fernet` with key-rotation support via a comma-separated `FIELD_ENCRYPTION_KEYS` env var. `MFASetting.secret` and `MFASetting.backup_codes` are now stored encrypted at rest; existing plaintext values are handled transparently on read (graceful `InvalidToken` fallback). Migration `20260408_0001_encrypt_sensitive_mfa_fields` converts the column types. A new `encrypt-existing-secrets` CLI command backfills pre-existing plaintext records idempotently. `FIELD_ENCRYPTION_KEYS` documented in `env.production.example`. Test suite added (`test_field_encryption.py`). **(2) SSP/POA&M ↔ Threat deep integration** — New `sync_scenario_controls_to_ssp()` service in `ssp/services.py` propagates CSA controls from a `ThreatScenario` into the linked SSP's minimum security controls on scenario create/edit. New `sync_mitigation_to_poam()` service automatically creates or updates a `POAMItem` in the linked SSP whenever a `ThreatMitigationAction` is created or edited; `POAMItem` gains a `source_threat_mitigation_id` FK (migration `20260409_0001_add_source_threat_mitigation_id_to_poam`). New `/ssp/<id>/controls/sync-from-threats` POST endpoint for manual bulk-sync of threat controls. `seed_controls()` now also walks direct and DPIA-linked threat models to seed controls from their scenarios. **(3) Risk dashboard & sync improvements** — `sync_from_threats` now also updates already-linked scenarios (not just creates new ones). `_components_for_scenario()` now first tries to match the scenario's asset name to a BIA component by name instead of falling back to the whole scope immediately. `configure_risk_form()` and `component_choices()` accept an optional `context_scope_id` to scope the component dropdown to the relevant system. Risk edit route re-syncs components from the current threat model on every load. Risk dashboard CSS refactored: duplicate nested class definitions removed, Bootstrap CSS variables replaced with app theme variables (`--color-muted`, `--color-border`), hero grid rewritten with CSS Grid, and `.risk-actions` / `.risk-card-grid` layout helpers extracted. |
| 2026-04-07 | Threat → Risk workspace synchronisation (2026.4.4 / 2026.4.4a) | Added a direct link between `ThreatScenario` and the Risk workspace: new `risk_id` FK on `ThreatScenario` (migration `20260407_0001_add_risk_id_to_threat_scenarios`) and a `sync_scenario_to_risk()` service in `threat/services.py`. Scenarios with `treatment=mitigate` and an active status are **automatically** synced to a Risk item on create/edit; if treatment changes to accept/transfer/avoid, or the scenario reaches a terminal status (mitigated/accepted/closed), the linked risk is automatically closed. Added a `/risk/sync-from-threats` POST endpoint for bulk-syncing all eligible unlinked scenarios (button on Risk dashboard for admins and assessment managers). Added a per-scenario `/threat/<id>/scenarios/<id>/sync-risk` endpoint and a "Sync now" button on the scenario detail page, which also shows a "Risk workspace" field with a linked badge and a link to the risk edit page. The risk `edit` route now auto-refreshes chance/impact/components from the linked scenario on load; a new `_resolve_components_for_edit()` helper allows saving risks that include components introduced via the threat-sync. New CLI command `backfill-threat-risks` (with `--dry-run`) for migrating pre-existing scenarios. 10 new translation keys in `en.json` and `nl.json`. Hotfix 2026.4.4a resolved an authorization regression in risk and threat routes introduced by the same release. |
